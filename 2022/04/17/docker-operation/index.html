1.安装docker

1.0. ubuntu安装：使用官方安装脚本自动安装

curl -fsSL https://get.docker.com | bash -s docker --mirror Aliyun

下面是debina版本：

1.1卸载旧版本

sudo apt remove docker docker-engine docker.io containerd runc

1.2安装先决条件

sudo apt install ca-certificates curl gnupg lsb-release

1.3添加密钥并加入官方仓库

Docker软件包由官方提供，因此还需要加入官方专用的Docker软件包源，这里推荐使用国内的中科大Docker软件包镜像源，执行如下命令添加：

sudo mkdir -p /etc/apt/keyrings
sudo curl -fsSL https://mirrors.ustc.edu.cn/docker-ce/linux/debian/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://mirrors.ustc.edu.cn/docker-ce/linux/debian $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

如果你的服务器位于中国大陆之外地区，还是建议使用官方下载源，那就按照下面命令添加：

sudo mkdir -p /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/debian/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/debian $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

1.4执行安装

sudo apt update
sudo apt install docker-ce docker-ce-cli containerd.io docker-compose-plugin

安装完成，执行下列命令，若能输出帮助信息说明安装成功：

docker -v

2.配置镜像仓库镜像源

前面介绍过了我们的镜像都是从官方的仓库拉取的，但是官方的仓库在国外，因此国内服务器最好是配置一下镜像地址为国内的镜像仓库地址。

我们这里使用阿里云的镜像加速器，然后修改Docker的配置文件/etc/docker/daemon.json，在配置文件中加入如下内容：

&#123;
    "registry-mirrors": ["https://v10rlwmg.mirror.aliyuncs.com"]
&#125;

然后执行下列命令重启Docker服务：

sudo systemctl daemon-reload
sudo systemctl restart docker

至此就配置完成了！

3.Docker引擎服务常用命令

安装完成Docker之后，默认Docker的守护进程已经运行起来了。日常若想操作这个守护进程可以通过下面的一些命令来完成。

3.1启动，停止和重启

# 启动Docker守护进程
sudo systemctl start docker
# 停止Docker守护进程
sudo systemctl stop docker
# 重启Docker守护进程
sudo systemctl restart docker

3.2开机自启动相关

# 设定Docker守护进程开机自启
sudo systemctl enable docker
# 禁用Docker守护进程开机自启
sudo systemctl disable docker

4.镜像的基本操作

# 查看镜像
docker images
# 搜索镜像
docker search 名字
# 拉取镜像
docker pull 镜像名
docker pull 镜像名:指定版本
# 删除镜像
docker rmi 镜像ID
# 删除全部镜像
docker rmi ${docker images -q}

5.容器基本操作

5.1容器创建和运行

我们已经拉取好了一个debian的镜像，我们就来创建一个Debian的容器：

docker run -it --name=myfirstdebian debian /bin/bash

参数说明：

  • i 表示交互式操作
  • t 使用终端，在前台运行
  • –name 给这个容器指定一个名字，这个非必须
  • debian 要使用的镜像名，这里使用debian镜像
  • /bin/bash 要在这个容器中执行的命令，通常接在镜像名后面，这里表示打开容器中Debian系统的bash终端

上面我们的容器是交互式的，在我们终端上运行，终端关闭容器也会停止，若想要容器后台运行，可以使用这个命令：

docker run -id --name=myseconddebian debian /bin/bash

把上述的t改成d，就表示在后台运行这个容器。这次没有直接进入容器，而是输出了容器的id。

所以说这俩个参数意义如下:

  • it 以交互式方式访问容器，容器在前台运行，终端退出容器也停止。
  • id 以守护式方式启动运行容器，容器在后台运行，终端退出容器仍然在运行。

5.2进入容器

对于正在后台运行的容器，可以通过docker exec进入。

docker exec -it myseconddebian /bin/bash
或 docker exec -it 2f7ee60376ba /bin/bash

这个时候就进入到了容器内部的终端了！不过这个时候使用exit退出，容器仍然在运行。

除此之外，我们还可以用attach来代替上述exec进入容器，但是使用attach的话，退出时容器也停止了。

5.3已存在的容器的启动和停止

上述docker run事实上是创建并启动容器。

# 启动容器
docker start 容器id
# 启动所有容器
docker start $(docker ps -aq)
# 停止和重启容器命令
docker stop ID/容器名
# 停止全部容器
docker stop $(docker ps -aq)
# 重启一个容器
docker restart ID/容器名
# 重启全部容器
docker restart $(docker ps -aq)

5.4删除已有容器

只能删除已经停止的容器

#删除容器
docker rm ID/容器名
#删除全部容器
docker rm $(docker ps -aq)

5.5查看容器信息

docker inspect ID/容器名

5.6容器的导入导出

如果说这个容器我想将其当前的状态作为一个快照备份一下，我们就可以把一个容器导出为一个tar文件，下次若这个容器出现了问题丢失了数据，我们还可以从这个tar文件恢复。

例如我要把ID为5a7cfd3e0dda的容器导出为文件debian-1.tar存放在/root目录下，通过一下命令：

docker export 5a7cfd3e0dda > /root/debian-1.tar

完成后，你就发现这个容器的当前状态被导出为一个tar文件，通过ls命令看到文件已保存，然后如果想把这个保存的状态导入，则执行：

docker import /root/debian-1.tar debian-recover

导入会把这个容器的状态作为一个镜像导入，执行上述命令后，你会发现多了名为debian-recover的镜像，然后再用docker run就可以启动这个镜像了！

6.数据卷

6.1配置数据卷

命令格式如下：

docker run ... -v 宿主机目录（文件）:容器内目录（文件）...

前面参数-it等等是一样的，主要是添加了-v参数，表示指定要配置的数据卷，一次可以挂载多个数据卷。

例如现在，创建一个名为d1的Debian容器，把本机目录/root/datavolume挂载到容器中/root/vol中：

docker run -id --name=d1 -v /root/datavolume:/root/vol debian /bin/bash

需要注意的是，路径必须都是绝对路径形式，并且不存在的目录都会自动创建。

数据卷就是把宿主机中一个目录挂载至容器中的一个目录中，俩边的目录任何一边发生变化都会同步。若挂载时原本目录有文件，挂载后容器中的对应目录也会有文件。

如果说要挂载多个目录则如下：

docker run ... -v 宿主机目录1:容器目录1 -v 宿主机目录2:容器目录2 ...

多写几个-v即可。

同样的，如果使俩个容器挂载同一个数据卷，这样可以实现容器间数据交换。

注意：数据卷只能在创建容器的时候设定！后续一个容器的数据卷不能增加或移除！

6.2数据卷容器

数据卷容器其实质上就是一个容器，只不过数据卷容器是专门用于数据交换来用的。也就是说，数据卷容器会挂载一个目录，然后另外多个容器可以挂载到这个数据卷容器上，然后宿主机的数据卷目录变化，数据卷的容器中对应目录也会变化，挂载数据卷容器的容器内容也会对应变化。这样也就相当于所有挂载数据卷容器的容器，间接地挂载了数据卷容器挂载的宿主机目录。并且即使数据卷容器出现问题停止了，也不会影响数据。

首先我们创建一个名为data的容器作为数据卷容器：

docker run -id --name=data -v /datavolume debian /bin/bash

可见这里创建数据卷容器时，-v后面的参数和前面的有所不同，前面是宿主机目录:容器中目录，现在只指定容器中目录，这样Docker会在宿主机上自动分配一个目录挂载至容器中的对应目录上。

然后再创建俩个容器名为c1和c2挂载到这个数据卷容器data上：

docker run -id --name=c1 --volumes-from data debian /bin/bash
docker run -id --name=c2 --volumes-from data debian /bin/bash

上述--volumes-from就代表指定要挂载的数据卷容器。

现在进入容器data，在其中的/datavolume中新建一个文件，然后退出并分别进入c1和c2容器下的/datavolume目录，你会发现刚刚数据卷中新建的文件也同步过来了！

我们通过docker inspect命令来查看上述的数据卷容器data中的/datavolume目录挂载载宿主机上的哪个目录。

docker inspect data

输出很多信息，找到Mounts字段,其中Source表示这个容器数据卷对应的宿主机目录，而Destination对应的容器内目录。

6.3docker cp命令

除了使用数据卷实现容器和宿主机的文件传输，还可以使用docker cp命令把容器内文件拷贝出来到宿主机，当然也可以反过来把宿主机内文件拷贝进容器。

# 把容器内文件拷贝到宿主机
docker cp 容器ID/容器名:容器内文件绝对路径 拷贝到宿主机的路径
# 把宿主机的文件拷进容器
docker cp 宿主机文件的路径 容器ID/容器名:拷贝到容器内的路径

例如：

# 把本机当前目录下的test文件拷贝到名为c1容器中的/root目录下
docker cp ./test c1:/root/
# 把本机当前目录下的test文件拷贝到ID为155ddba25935容器中的/root目录下并重命名为test-file
docker cp ./test 155ddba25935:/root/test-file
# 把名为c1容器中文件/root/test拷贝到本机当前路径下并重命名为tt
docker cp c1:/root/test ./tt
# 把本机当前目录下的datavolume目录拷贝到名为c1容器中/root目录下
docker cp ./datavolume/ c1:/root/

可见无论是文件还是目录都可以拷贝，通常目录后以/结尾。

7.Docker网络

7.1Docker内网

事实上，Docker所有的容器会在宿主机上构成一个内网，这个是存在于宿主机上的专属于所有Docker容器的内网。其中：

  • 所有的容器可以通过内网互相访问
  • 宿主机可以通过内网访问每一个容器对应的端口
  • 但是外网是不能直接访问某个宿主机中某个容器的端口

7.2端口映射

比如说你把MySQL容器的3306端口映射到宿主机的3307端口上，容器启动后，外网访问宿主机的3307端口，就可以访问到MySQL容器的3306端口上。

现在我们以MySQL镜像为例，在服务器上运行一个MySQL容器，并把其3306端口映射至宿主机3307端口。

docker run -id --name=mysql1 -v /root/mysql/conf:/etc/mysql/conf.d -v /root/mysql/logs:/logs -v /root/mysql/data:/var/lib/mysql -p 3307:3306 -e MYSQL_ROOT_PASSWORD=123456 mysql

参数说明：

  • 进行了三个目录的挂载：
    • 把容器的/etc/mysql/conf.d目录挂载至本机的/root/mysql/conf目录，这是MySQL的配置文件目录，便于我们修改配置文件
    • 把容器的/logs目录挂载至本机的/root/mysql/logs目录，便于我们查看日志
    • 把容器的/var/lib/mysql目录挂载至本机的/root/mysql/data目录，/var/lib/mysql目录是MySQL的数据文件目录，挂载至本机可以便于容器出现异常后还可以从本机恢复数据
  • -p 3307:3306把容器的3306端口映射到本机3307端口
  • -e MYSQL_ROOT_PASSWORD=123456MySQL容器的一个参数，表示初始化MySQL的root用户密码为123456，-e表示设定容器中系统的环境变量。

可见，-p参数值的格式是本机端口:容器端口，这样容器的对应端口就被映射到本机端口了！

7.3Docker内网的虚拟路由

Docker内网中有一个虚拟的路由器，这个虚拟的路由器也就是用于给每个容器分配内网ip地址。当然容器分配的内网ip地址只能是容器和容器之间访问或者是宿主机访问容器，外网不能直接访问，除非是通过上述端口映射。

我们默认所有的容器，都会分配给一个名为bridge的虚拟路由器来分配ip地址。

7.3.1查看一个路由内的所有容器及其IP

# 查看名为bridge的虚拟路由器下的所有容器
docker network inspect bridge

找到Containers字段即可看到这个虚拟路由下所有的容器。

7.3.2创建新的路由器

虽然大多数情况我们使用默认的虚拟路由bridge即可，不过我们仍然可以创建多个虚拟路由，并把容器连接到不同的路由下。

通过以下命令查看现在所有的虚拟路由列表：

docker network ls

现在如果要创建一个名为cfl的虚拟路由，命令如下：

docker network create cfl

我们可以把一个已有的容器连接到指定的虚拟路由，例如现在我要把mysql1容器连接到路由cfl下，命令如下：

# 命令格式：docker network connect 路由名 容器名/ID
docker network connect cfl mysql1

与此同时，在创建容器的时候，我们可以加上--network参数指定创建这个容器时将其连接到哪个虚拟路由器：

# 创建一个名为nginx3的Nginx容器，并连接到虚拟路由器cfl下
docker run -id --name=nginx3 --network=cfl nginx

注意，位于不同的虚拟路由下的容器是不能互相访问的，也就是说虚拟路由通常是为了隔离一些容器的网络环境。

7.3.3Docker网络常用命令

# 查看帮助
docker network --help
# 把名为nginx3的容器从虚拟路由cfl下断开连接，此时这个容器处于离线状态无法通过内网访问
docker network disconnect cfl nginx3
# 删除名为cfl的虚拟路由器，需要将路由器下的全部容器断开连接才能删除
docker network rm cfl

8.Docker镜像制作

8.1进一步认识Docker镜像

8.1.1Linux系统文件组成

首先，我们需要了解一下Linux的文件系统，其文件系统由俩部分组成：

  1. bootfs包含了俩部分：引导加载程序bootloader和系统内核kernel
  2. rootfs root文件系统，包含系统基本目录结构如：/dev,/bin,/etc等等

对于不同的Linux发行版，其bootfs基本上是一样的，而rootfs不一样。

8.1.2Docker镜像可叠加性

Docker的镜像事实上是由特殊的文件系统叠加而成，也就是说，一个镜像可能由多个镜像叠加。

镜像的最低端是bootfs,但是使用的是宿主机的bootfs,这一层不包含在镜像中。这说明不同的Docker镜像都复用了宿主机的内核。

第二层则是rootfs,即为基础镜像，然后再往上就可以叠加其他的镜像文件。

细心的同学也发现了，在拉取镜像的时候，是一层一层地拉取的，这正是因为镜像都是叠加而来的。

所以说，一个Debian的镜像事实上只是包含了Debian操作系统的rootfs部分，并没有内核，启动引导和驱动等等，这些都是使用的宿主机上的，所以说一个Debian镜像很小。

而应用程序的镜像都是基于系统rootfs基础镜像进一步叠加上应用的环境和应用程序文件制作的，因此一个Nginx镜像中不仅仅包含了Nginx，还包含了一个rootfs基础镜像，这样体积就会大一些。

一个镜像也可以放在另一个镜像的上面，下面的镜像就成为父镜像，最底部的镜像称为基础镜像。

所以说，Windows系统上是不能安装Linux镜像的，因为俩者内核不一样，无法复用。

镜像都是只读的，若要修改需要用镜像创建容器再来修改其中的文件，然后容器可以再转成镜像。

用镜像创建容器时，事实上是在这个只读镜像上进一步放了一层称之为可写容器，即为我们之前操作的容器。然后这一起又可以做成一个新的镜像给别人用，做成新的镜像之后，你之前的可写容器的操作，就是在原有镜像上又加了一层镜像。

所以说，Docker的镜像的本质是一个分层的文件系统，分层就是便于复用。

通过docker inspect命令还可以看一个镜像的信息，比如其镜像层：

# 查看mysql镜像的信息
docker inspect mysql

找到RootFS中Layers即可看到这个镜像包含多少层。后面我们将学习如何制作镜像，制作镜像通常有俩种方式：把已有的容器转换成镜像或者是通过编写Dockerfile。

8.2容器转镜像

比如说我们要部署一个网页应用，我们可以用Nginx的镜像创建一个容器，然后在容器中将Nginx配置好，并配置好静态网页之后，再把这个容器转换成镜像。

通过docker commit实现容器转镜像：

docker commit 容器名/ID 自定义镜像名:版本号

这个时候，你的容器就转换成镜像了！除此之外，我们还可以把镜像转换成压缩文件，共享给别人，别人就可以用你的镜像直接部署应用。

镜像的导入导出命令：

# 把镜像导出成压缩文件
docker save -o 压缩文件名 镜像名/ID:版本号
# 把压缩文件导入成镜像
docker load -i 压缩文件名

上述版本号部分可以省略，省略后就默认代表为最新版本latest。

需要注意的是，容器挂载的数据卷，在commit的时候不会被包含到镜像中去。

8.3Dockerfile

Dockerfile事实上只是一个文本形式的文件，但是Docker可以根据这个文件去构建一个镜像。

8.3.1Dockerfile指令

  • FROM指令表示基于哪个镜像开始构建，scratch是空镜像。
  • ADD指令表示构建镜像的时候把指定文件添加到镜像中某个位置，若被指定文件为可识别的压缩文件（tar,gz,bzip2或者xz），则会被解压到镜像中目的位置，这个命令后面可以指定本地文件也可以是远程URL文件。
  • COPY表示构建镜像的时候把指定文件复制到镜像中，和ADD不同，它不会自动解压文件，也不能从远程URL下载。
  • CMD指令表示启动容器时，若没有指定命令，则默认执行这个CMD后面的命令，一个Dockerfile仅仅最后一个CMD起作用，命令格式通常是JSON数组形式，第一个是命令，后面都是参数，例如默认命令是7z a -t7z a.7z ./*则写成CMD ["7z","a","-t7z","a.7z","./*"]，也可以直接写成普通命令形式7z a -t7z a.7z ./*,当然推荐使用JSON数组形式。

之前通过docker run启动Debian镜像在后面我们指定执行了/bin/bash命令以进入容器中的命令行，不过事实上你不加这个指令也可以，因为Dockerfile中指定了CMD即默认命令，启动容器时不指定命令就会执行这个默认命令。

8.3.2通过Dockerfile构建镜像

我想把自己的springboot项目部署到Docker并制作成容器。

FROM java:8-alpine
ADD user-center-0.0.1-SNAPSHOT.jar user-center.jar
CMD ["java","-jar","user-center.jar"]

编写完成，把Dockerfile和你的jar文件一起上传至服务器，由于上述Dockerfile中使用的是相对路径，因此jar文件和Dockerfile文件的相对路径应当保持一致，即都在同一目录下。上传后也得保持相对路径一致。

上传后进入对应目录，执行构建命令：

docker build -f dockerfile文件路径 -t 镜像名:版本 构建时路径

例如我进行构建：

docker build -f test.dockerfile -t test:1.0 .
我写的简略版本 docker build -t test:1.0 .

版本号不写则为最新latest。

上述命令：

  • -f指定Dockerfile文件
  • -t指定构建后镜像名和版本号
  • 构建时路径在Dockerfile中ADD或者COPY指令添加文件时采用的相对路径就是相对于这个构建时路径的，例如上述ADD user-center-0.0.1-SNAPSHOT.jar user-center.jar,然后指定构建路径为/root的话，构建镜像添加文件时就会找/root/user-center-0.0.1-SNAPSHOT.jar并加入到镜像，不过这个通常写成.表示当前路径即可。

最后显示successful字样则成功。

最后我们把它跑起来：

docker run --name test -p8080:8080 -d test:1.0
